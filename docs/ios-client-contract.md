# Linea iOS Client Contract

This document defines the v1 contract between the future native iOS client and the local Linea server.

## Setup

The iOS client should collect and persist two local settings:

- Server URL, for example `http://<dev-machine-lan-ip>:8787`.
- Server bearer token, generated by the Linea server on first database initialization.

After the user taps Save, validate the configuration by calling:

```http
GET /auth/check HTTP/1.1
Host: <configured-server-host>
Authorization: Bearer <server-token>
```

A successful validation returns `200 OK` with:

```json
{"ok": true}
```

If validation returns `401 Unauthorized` or the request cannot reach the server, keep the app in `unconfigured` or `error` state and ask the user to correct the settings.

## Authentication

Every protected request must include this header format:

```http
Authorization: Bearer <server-token>
```

The token is a Linea server token, not an xAI API key. The iOS client must never receive, store, display, or transmit `XAI_API_KEY`; xAI credentials stay server-side only.

## Call start

To start a call:

1. The iOS client creates a local WebRTC peer connection.
2. The iOS client creates an SDP offer.
3. The iOS client sends the offer to the Linea server:

```http
POST /webrtc/offer HTTP/1.1
Host: <configured-server-host>
Authorization: Bearer <server-token>
Content-Type: application/json
```

Request body:

```json
{
  "type": "offer",
  "sdp": "v=0..."
}
```

4. The server returns an SDP answer:

```json
{
  "type": "answer",
  "sdp": "v=0...",
  "call_id": "..."
}
```

5. The iOS client applies the SDP answer to the local WebRTC peer connection.

V1 supports one active call at a time. If another call is already active, `POST /webrtc/offer` returns `409 Conflict`; the client should show an error and stay or return to `idle` when safe.

## Call stop

To stop a call in v1, close the local WebRTC peer connection. The server also has an idle timeout and may end an inactive call after no media/events flow for the configured timeout.

There is no separate v1 stop endpoint.

If the user starts again immediately after Stop and the server returns `409 Conflict`, keep or return the client to `idle`, show that the previous call is still being cleaned up server-side, and ask the user to retry manually after a short wait. Do not auto-reconnect.

## UI state machine

The client should model these states:

- `unconfigured`: server URL or bearer token is missing or has not passed validation.
- `idle`: configuration is valid and no call is active.
- `connecting`: Start was tapped; WebRTC offer creation and `POST /webrtc/offer` are in progress.
- `active`: SDP answer was applied and the peer connection is active.
- `stopping`: Stop was tapped; the peer connection is closing.
- `error`: configuration, auth, network, SDP, or call setup failed.

V1 behavior:

- `connecting` is disabled/not cancellable. The Start/Stop control should not allow a second action until setup succeeds or fails.
- `active` exposes Stop.
- `stopping` disables Start/Stop until local cleanup completes.
- `error` should display a concise message and offer a manual retry or return to setup depending on the failure.

## Reconnect policy

Do not auto-reconnect in v1. If networking, authentication, WebRTC setup, or server-side call handling fails, the client should surface the failure and require the user to manually retry.

## Local HTTP and ATS caveat

The v1 Linea server uses plain HTTP for local development only. A native iOS development build may need an App Transport Security exception for the configured local server host or LAN IP.

Keep any ATS exception as narrow as possible. Do not ship a broad production ATS bypass, and do not expose the local Linea HTTP server to the public internet.
